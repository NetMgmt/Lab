# Deliverables Directory

This directory is intended to store **all files that are considered deliverables** for the laboratory practice, such as reports, generated configuration files, logs, screenshots, scripts, or any other artifacts that must be submitted.

The contents of this directory can be made available inside the Docker containers by mounting it as a **bind mount** in the `compose.yml` file.

## Example

The following example mounts the `deliverables` directory from the host into the `manager` container:

```yaml
services:
  manager:
    volumes:
      - ./deliverables:/home/kenobi/deliverables
```

The same approach can be used for any other container that needs access to the deliverable files.

## Organizing Deliverables per Host

For scenarios where each container requires its own set of files, it is recommended to organize the directory using one subdirectory per host. For example:

```text
deliverables/
├── manager/
├── host-1/
└── host-2/
```

Each container can then mount its corresponding subdirectory:

```yaml
services:
  manager:
    volumes:
      - ./deliverables/manager:/home/kenobi/deliverables

  host-1:
    volumes:
      - ./deliverables/host-1:/home/kenobi/deliverables

  host-2:
    volumes:
      - ./deliverables/host-2:/home/kenobi/deliverables
```

This organization keeps the files generated by each host separated and simplifies their management, especially in scenarios involving multiple machines.

## Mount Point Inside the Containers

Regardless of the organization used on the host system, the mounted files will always be available inside each container at:

```text
/home/kenobi/deliverables
```

This provides a consistent location for laboratory scripts and documentation to read from or write to.

## Linux Host Permissions

When running Docker on a Linux host, it is recommended to grant read, write, and traversal permissions to all users for the `deliverables` directory before starting the containers:

```bash
chmod -R a+rwX deliverables
```

This ensures that the containers have full access to the mounted files and directories, preventing permission-related issues during the execution of the laboratory exercises.
